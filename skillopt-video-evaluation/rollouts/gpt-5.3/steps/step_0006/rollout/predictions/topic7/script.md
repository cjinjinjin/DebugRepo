# OpenClaw 视频口播稿（聚焦版）

这轮 OpenClaw 爆火，核心不是又多了一个聊天界面，而是它把“会回答”推进到了“会持续执行”。

文中给出的热度信号很强：微信指数破亿，项目短时间冲到 297K Stars、116K+ 社区规模、1000+ 贡献者。

但真正决定上限的，不是热度，而是架构链路：Inference、Memory、RAG、MCP、Skills。

Inference 负责把模型跑起来；Memory 负责补上无状态缺口；RAG 负责把知识从“冻结参数”变成“可检索更新”；MCP 负责接工具；Skills 负责把动作编排成可复用流程。

真正关键是 MCP 和 Skills 的组合：一个解决“能不能调用”，一个解决“按什么顺序持续执行”。

部署阶段的正确顺序是：先把环境、安装、初始化、链路联调跑通，再做参数精调。

参数主要盯四个：temperature、top_p、penalty、max_tokens。

工程上一定要有 fallback：主模型异常时自动切备份，任务链才不会中断。

接入飞书的价值在于：把 Agent 放进团队正在使用的沟通入口，减少额外学习与切换成本。

典型流程是六步：建应用、开权限、发布、配置通道、配置事件订阅、联调。

然后是技能系统。技能不是插件收藏夹，而是高频任务的 SOP 模块化。

最后必须收束到安全：能力给得越多，边界越要硬。

最小权限要默认开启；支付、删除、安装、外发数据要显式确认；外部插件和输出要做供应链审计；密钥与敏感目录要有硬治理。

这套方法最终落到真实场景：晨报、邮件分流、手机 DevOps、知识库问答、第二大脑。

关键不是“能不能做一件事”，而是“能不能每天稳定完成一串事”。

结尾只留一句行动建议：先跑通基础链路，再逐步放权，不要反过来。
